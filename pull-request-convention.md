# Pull Request 작성 지침

## 1. PR 기본 목적

Pull Request는 코드 변경 사항을 팀원에게 공유하고, 리뷰를 통해 코드 품질을 높이기 위한 과정입니다.

PR에는 단순히 변경 결과만 작성하는 것이 아니라, 리뷰어가 변경 내용을 빠르게 이해할 수 있도록 다음 내용을 함께 작성합니다.

* 어떤 작업을 했는지
* 왜 작업했는지
* 어떤 부분이 변경되었는지
* 어떻게 테스트했는지
* 리뷰어가 어떤 부분을 중점적으로 보면 좋은지
* 관련 이슈나 참고 자료가 있는지

---

## 2. PR 제목 작성 형식

PR 제목은 커밋 메시지 규칙과 동일하게 작성합니다.

형식:

Type: Subject

필요한 경우 변경 범위를 나타내기 위해 Scope를 사용할 수 있습니다.

형식:

Type(scope): Subject

예시:

Feat: 회원 가입 기능 구현

Fix: 로그인 실패 오류 수정

Docs: README 설치 방법 추가

Refactor: 사용자 조회 로직 개선

Feat(auth): JWT 로그인 기능 추가

Fix(user): 사용자 조회 조건 오류 수정

---

## 3. PR 제목 작성 규칙

| 규칙               | 설명                                          |
| ---------------- | ------------------------------------------- |
| Type 사용          | 변경 목적에 맞는 Type을 사용합니다.                      |
| 제목은 간결하게 작성      | PR의 핵심 변경 내용을 한눈에 알 수 있게 작성합니다.             |
| 제목 끝에 마침표 사용 안 함 | 제목 끝에 마침표를 붙이지 않습니다.                        |
| 모호한 제목 지양        | “수정”, “작업”, “개선”처럼 범위가 불명확한 표현만 사용하지 않습니다.  |
| 필요 시 Scope 사용    | 변경 범위가 명확하면 Type(scope): Subject 형식을 사용합니다. |
| 한 가지 목적 표현       | 하나의 PR 제목에는 하나의 핵심 목적만 담습니다.                |

## 4. PR Type 목록

| Type             | 의미                              |
| ---------------- | ------------------------------- |
| Feat             | 새로운 기능 추가                       |
| Fix              | 버그 수정                           |
| Design           | CSS 등 사용자 UI 디자인 변경             |
| !BREAKING CHANGE | 큰 API 변경                        |
| !HOTFIX          | 급하게 치명적인 버그 수정                  |
| Style            | 코드 포맷 변경, 세미콜론 누락 등 코드 의미 변경 없음 |
| Refactor         | 프로덕션 코드 리팩토링                    |
| Comment          | 주석 추가 또는 변경                     |
| Docs             | 문서 수정                           |
| Test             | 테스트 코드 추가 또는 수정, 프로덕션 코드 변경 없음  |
| Chore            | 빌드, 패키지 매니저, 설정 변경 등            |
| Rename           | 파일 또는 폴더명 변경/이동                 |
| Remove           | 파일 삭제                           |

## 5. PR 본문 기본 구조

PR 본문은 다음 항목을 기준으로 작성합니다.

## 작업 내용

이번 PR에서 작업한 내용을 요약합니다.

예시:

* 회원 가입 API를 추가했습니다.
* 이메일 중복 확인 로직을 추가했습니다.
* 회원 가입 요청 DTO 검증 조건을 추가했습니다.

## 작업 이유

이 작업이 필요한 이유를 작성합니다.

예시:

회원 가입 시 동일한 이메일로 중복 계정이 생성될 수 있어, 가입 요청 전에 이메일 중복 여부를 확인하도록 개선했습니다.

## 변경 사항

실제로 변경된 기능, 로직, 화면, 설정 등을 작성합니다.

예시:

* 회원 가입 API 추가
* 이메일 중복 확인 서비스 로직 추가
* 회원 가입 요청 값 검증 추가
* 중복 이메일 요청 시 예외 응답 처리 추가

## 테스트 방법

어떤 방식으로 테스트했는지 작성합니다.

예시:

* Postman으로 회원 가입 API 요청 확인
* 중복 이메일 요청 시 오류 응답 확인
* 정상 회원 가입 요청 시 사용자 데이터 저장 확인
* 회원 가입 관련 테스트 코드 실행 확인

## 리뷰 요청 사항

리뷰어가 중점적으로 확인해야 할 부분을 작성합니다.

예시:

* 이메일 중복 확인 로직 위치가 적절한지 확인 부탁드립니다.
* 예외 메시지와 응답 코드가 현재 API 규칙과 맞는지 확인 부탁드립니다.
* DTO 검증 조건이 요구사항을 충분히 반영하는지 확인 부탁드립니다.

## 관련 이슈

관련된 이슈가 있을 경우 작성합니다.

예시:

Resolves: #123

Ref: #456

Related to: #48, #45

## 7. PR 작성 시 포함하면 좋은 내용

PR에는 변경 사항에 따라 다음 내용을 포함하면 좋습니다.

* 주요 변경 사항
* 작업 이유
* 테스트 방법
* 리뷰 요청 사항
* 관련 이슈
* 화면 변경이 있다면 스크린샷 또는 GIF
* API 변경이 있다면 요청/응답 예시
* DB 변경이 있다면 테이블, 컬럼, 마이그레이션 정보
* 설정 변경이 있다면 환경 변수나 설정 파일 변경 내용
* 배포 시 주의사항

## 8. PR 작성 시 피해야 할 내용

다음과 같은 PR 작성은 피합니다.

### 제목이 너무 모호한 경우

나쁜 예시:

수정

작업 완료

버그 고침

리팩토링

좋은 예시:

Fix: 로그인 실패 시 오류 메시지 수정

Refactor: 사용자 조회 로직 분리

### 변경 범위가 너무 넓은 경우

나쁜 예시:

Feat: 로그인 구현 및 README 수정과 코드 정리

좋은 예시:

Feat: 로그인 API 구현

Docs: 로그인 API 문서 추가

Refactor: 로그인 검증 로직 분리

### 본문 설명이 부족한 경우

나쁜 예시:

작업했습니다.

좋은 예시:

* 로그인 실패 시 동일한 오류 메시지만 반환되던 문제를 수정했습니다.
* 실패 원인에 따라 적절한 응답 코드를 반환하도록 변경했습니다.
* Postman으로 실패 케이스별 응답을 확인했습니다.

## 9. PR 단위 원칙

PR은 가능한 한 하나의 목적만 가지도록 작성합니다.

* 기능 추가 PR
* 버그 수정 PR
* 리팩토링 PR
* 문서 수정 PR
* 설정 변경 PR

서로 다른 목적의 변경은 가능하면 PR을 분리합니다.

좋은 예시:

Feat: 로그인 API 구현

Fix: 로그인 실패 시 오류 메시지 수정

Docs: 로그인 API 문서 추가

나쁜 예시:

Feat: 로그인 구현 및 README 수정과 코드 정리

---

## 10. PR 크기 기준

리뷰하기 좋은 PR은 변경 범위가 명확하고 크기가 너무 크지 않아야 합니다.

| 기준            | 설명                                   |
| ------------- | ------------------------------------ |
| 작은 PR 권장      | 리뷰어가 빠르게 이해할 수 있도록 변경 범위를 줄입니다.      |
| 큰 PR은 설명 보강   | 변경 범위가 크다면 작업 배경과 리뷰 포인트를 자세히 작성합니다. |
| 관련 없는 변경 제외   | 포맷팅, 리팩토링, 기능 추가를 한 PR에 섞지 않습니다.     |
| 리뷰 가능한 단위로 분리 | 기능별, 모듈별, 목적별로 PR을 나눕니다.             |

---

## 11. 테스트 방법 작성 규칙

테스트 방법은 리뷰어가 직접 확인할 수 있도록 구체적으로 작성합니다.

좋은 예시:

* Postman으로 POST /api/users 요청 확인
* 중복 이메일 입력 시 409 응답 확인
* 정상 요청 시 users 테이블에 데이터 저장 확인
* 회원 가입 단위 테스트 통과 확인

나쁜 예시:

* 테스트 완료
* 확인함
* 문제없음

---

## 12. 리뷰 요청 사항 작성 규칙

리뷰 요청 사항에는 리뷰어가 집중해서 봐야 할 내용을 작성합니다.

예시:

* 예외 처리 방식이 기존 API 응답 규칙과 맞는지 확인 부탁드립니다.
* 서비스 계층에 중복 확인 로직을 두는 방식이 적절한지 확인 부탁드립니다.
* DTO 검증 조건이 요구사항을 충분히 반영하는지 확인 부탁드립니다.
* 메서드 분리 기준이 과하지 않은지 확인 부탁드립니다.

---

## 13. 화면 변경이 있는 경우

화면 변경이 있는 PR은 가능하면 스크린샷이나 GIF를 첨부합니다.

작성 예시:

## 화면 변경

* 로그인 버튼 위치를 중앙 정렬로 변경했습니다.
* 입력 필드 간격을 조정했습니다.
* 모바일 화면에서 버튼이 잘리는 문제를 수정했습니다.

## 확인 화면

* 변경 전 화면
* 변경 후 화면

---

## 14. API 변경이 있는 경우

API 변경이 있는 PR은 요청/응답 변경 내용을 작성합니다.

포함하면 좋은 내용:

* API 경로
* HTTP Method
* 요청 파라미터
* 요청 Body
* 응답 Body
* 응답 코드
* 에러 응답
* 기존 API와 달라진 점

작성 예시:

## API 변경 사항

* POST /api/users API 추가
* 이메일 중복 시 409 응답 반환
* 요청 Body에 email, password, name 필드 추가
* 회원 가입 성공 시 사용자 ID 반환

---

## 15. DB 변경이 있는 경우

DB 변경이 있는 PR은 테이블, 컬럼, 인덱스, 마이그레이션 정보를 작성합니다.

포함하면 좋은 내용:

* 추가된 테이블
* 변경된 컬럼
* 삭제된 컬럼
* 인덱스 추가 여부
* 마이그레이션 필요 여부
* 기존 데이터 영향 여부

작성 예시:

## DB 변경 사항

* users 테이블에 phone_number 컬럼 추가
* email 컬럼에 unique 제약 조건 추가
* 기존 사용자 데이터에는 영향 없음

---

## 16. 배포 시 주의사항

배포할 때 추가 작업이 필요한 경우 반드시 작성합니다.

예시:

* 환경 변수 JWT_SECRET 추가 필요
* 운영 DB 마이그레이션 필요
* 배포 전 캐시 초기화 필요
* 기존 API 응답 형식 변경으로 프론트엔드 반영 필요

---

## 17. PR 체크리스트

PR을 생성하기 전에 아래 항목을 확인합니다.

* [ ] PR 제목이 변경 내용을 명확히 설명하는가?
* [ ] PR 본문에 작업 내용이 작성되어 있는가?
* [ ] 작업 이유가 작성되어 있는가?
* [ ] 변경 사항이 정리되어 있는가?
* [ ] 테스트 방법이 작성되어 있는가?
* [ ] 리뷰 요청 사항이 작성되어 있는가?
* [ ] 관련 이슈가 있다면 연결되어 있는가?
* [ ] 불필요한 변경 사항이 포함되어 있지 않은가?
* [ ] 한 PR에 하나의 목적만 담겨 있는가?
* [ ] 화면 변경이 있다면 스크린샷을 첨부했는가?
* [ ] API 변경이 있다면 요청/응답 변경 내용을 작성했는가?
* [ ] DB 변경이 있다면 마이그레이션 정보를 작성했는가?
* [ ] 배포 시 주의사항이 있다면 작성했는가?
