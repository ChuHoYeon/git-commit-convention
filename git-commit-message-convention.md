# Git Commit Message Convention

## 1. 커밋 메시지 기본 구조

커밋 메시지는 보통 다음 3가지 영역으로 구성한다.

```text
Type: Subject

Body

Footer
````

* **Subject**는 필수이다.
* **Body**와 **Footer**는 선택사항이다.
* 각 영역은 빈 줄로 구분한다.
* 단순한 변경이라면 제목만 작성해도 된다.
* 변경 이유나 상세 설명이 필요할 경우 Body와 Footer를 추가한다.

---

## 2. 제목 작성 형식

커밋 제목은 다음 형식을 따른다.

```text
Type: Subject
```

예시:

```text
Feat: 회원 가입 기능 구현
Fix: 로그인 실패 오류 수정
Docs: README 설치 방법 추가
Refactor: 사용자 조회 로직 개선
```

---

## 3. 커밋 타입

| Type             | 의미                              |
| ---------------- | ------------------------------- |
| Feat             | 새로운 기능 추가                       |
| Fix              | 버그 수정                           |
| Design           | CSS 등 사용자 UI 디자인 변경             |
| !BREAKING CHANGE | 큰 API 변경                        |
| !HOTFIX          | 급하게 치명적인 버그 수정                  |
| Style            | 코드 포맷 변경, 세미콜론 누락 등 코드 의미 변경 없음 |
| Refactor         | 프로덕션 코드 리팩토링                    |
| Comment          | 주석 추가 또는 변경                     |
| Docs             | 문서 수정                           |
| Test             | 테스트 코드 추가 또는 수정, 프로덕션 코드 변경 없음  |
| Chore            | 빌드, 패키지 매니저, 설정 변경 등            |
| Rename           | 파일 또는 폴더명 변경/이동                 |
| Remove           | 파일 삭제                           |

---

## 4. 제목 작성 규칙

제목은 커밋의 핵심을 짧고 명확하게 표현한다.

| 규칙                  | 설명                        |
| ------------------- | ------------------------- |
| 50자 이내로 작성          | 제목은 너무 길지 않게 작성한다.        |
| Type 뒤에 콜론 사용       | `Type: 제목` 형식을 사용한다.      |
| 콜론 뒤에 공백 1칸 사용      | `Feat: 기능 추가`처럼 작성한다.     |
| 제목 끝에 마침표 사용 안 함    | 제목 끝에 `.`을 붙이지 않는다.       |
| 특수기호 사용 최소화         | 불필요한 특수기호는 사용하지 않는다.      |
| 간결하게 작성             | 변경 내용을 한눈에 알 수 있게 작성한다.   |
| 개조식 구문으로 작성         | 완전한 문장보다 요점 중심으로 작성한다.    |
| 영문 작성 시 동사 원형 사용    | Add, Fix, Update 등을 사용한다. |
| 영문 작성 시 과거시제 사용 안 함 | Added, Fixed처럼 쓰지 않는다.    |

---

## 5. Scope 사용 규칙

필요한 경우 Type 뒤에 괄호를 사용해 변경 범위를 표시할 수 있다.

```text
Type(scope): Subject
```

예시:

```text
Feat(auth): 로그인 기능 추가
Fix(db): 사용자 조회 오류 수정
Docs(readme): 설치 방법 추가
Refactor(user): 사용자 조회 로직 개선
```

Scope는 변경된 도메인, 기능, 모듈, 파일 범위 등을 나타낼 때 사용한다.

---

## 6. 본문 작성 규칙

본문은 커밋 제목만으로 설명이 부족할 때 작성한다.

본문에는 주로 다음 내용을 작성한다.

* 무엇을 변경했는지
* 왜 변경했는지

본문에는 가능하면 다음 내용은 길게 작성하지 않는다.

* 어떻게 변경했는지

구현 방식은 실제 코드에서 확인할 수 있으므로, 본문에는 변경 목적과 이유를 중심으로 작성한다.

예시:

```text
Feat: 회원 가입 기능 구현

회원 가입 시 이메일 중복 여부를 확인하도록 API를 추가했다.
중복 계정 생성을 방지하기 위해 가입 요청 전에 검증 로직을 수행한다.
```

---

## 7. 꼬리말 작성 규칙

꼬리말은 선택사항이며, 이슈 트래커 ID를 작성할 때 사용한다.

```text
유형: #이슈번호
```

예시:

```text
Resolves: #123
Ref: #456
Related to: #48, #45
```

---

## 8. 꼬리말 유형

| 유형         | 의미                     |
| ---------- | ---------------------- |
| Fixes      | 이슈 수정 중, 아직 해결되지 않은 경우 |
| Resolves   | 이슈를 해결한 경우             |
| Ref        | 참고할 이슈가 있는 경우          |
| Related to | 관련 이슈가 있는 경우           |

---

## 9. 전체 커밋 메시지 예시

```text
Feat: 회원 가입 기능 구현

SMS와 이메일 중복 확인 API를 추가하여 회원 가입 시 중복 계정 생성을 방지하도록 개선했다.

Resolves: #123
Ref: #456
```

---

## 10. 짧은 커밋 메시지 예시

```text
Feat: 회원 가입 기능 구현
```

```text
Fix: 로그인 실패 오류 수정
```

```text
Docs: README 설치 방법 추가
```

```text
Chore: Gradle 설정 변경
```

```text
Refactor: 사용자 조회 로직 개선
```

```text
Test: 회원 가입 테스트 코드 추가
```

---

## 11. 자주 사용하는 영어 동사

| 표현        | 사용 상황              |
| --------- | ------------------ |
| Fix       | 버그 수정              |
| Update    | 기존 기능 보완           |
| Add       | 기능, 문서, 예시 등 추가    |
| Remove    | 불필요한 코드나 파일 제거     |
| Refactor  | 코드 구조 개선           |
| Simplify  | 코드 단순화             |
| Improve   | 성능, 접근성, 검증 등 개선   |
| Implement | 기능 구현              |
| Correct   | 문법, 타입, 이름 등 오류 수정 |
| Prevent   | 특정 동작을 막음          |
| Avoid     | 특정 상황을 회피하도록 처리    |
| Move      | 코드나 파일 이동          |
| Rename    | 이름 변경              |

---

## 12. 커밋 단위 원칙

한 커밋에는 한 가지 변경 사항만 담는다.

* 변경 이력을 추적하기 쉽게 유지한다.
* 너무 많은 변경 사항을 한 커밋에 담지 않는다.
* 기능 추가, 버그 수정, 리팩토링, 문서 수정은 가능하면 분리한다.
* 하나의 커밋이 하나의 목적을 가지도록 작성한다.

좋은 예시:

```text
Feat: 로그인 API 구현
```

```text
Fix: 로그인 실패 시 오류 메시지 수정
```

나쁜 예시:

```text
Feat: 로그인 구현 및 README 수정과 코드 정리
```

---

## 13. 커밋 메시지 작성 예시 모음

### 기능 추가

```text
Feat: 회원 가입 기능 구현
```

```text
Feat(auth): JWT 로그인 기능 추가
```

### 버그 수정

```text
Fix: 로그인 실패 오류 수정
```

```text
Fix(db): 사용자 조회 조건 오류 수정
```

### 문서 수정

```text
Docs: README 설치 방법 추가
```

### 리팩토링

```text
Refactor: 사용자 조회 로직 개선
```

### 설정 변경

```text
Chore: Gradle 설정 변경
```

### 테스트 추가

```text
Test: 회원 가입 테스트 코드 추가
```

---

## 14. 커밋 메시지 작성 체크리스트

커밋하기 전에 아래 항목을 확인한다.

* [ ] Type이 변경 내용과 맞는가?
* [ ] 제목이 50자 이내인가?
* [ ] `Type: Subject` 형식을 지켰는가?
* [ ] 제목 끝에 마침표를 붙이지 않았는가?
* [ ] 한 커밋에 한 가지 변경 사항만 담았는가?
* [ ] 설명이 필요한 경우 Body를 작성했는가?
* [ ] 관련 이슈가 있다면 Footer를 작성했는가?
